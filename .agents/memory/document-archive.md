---
name: Архив прикреплённых документов
description: Где и как хранятся оригиналы загружаемых файлов и почему отметка «закреплён» едина для всех
---

Оригиналы загружаемых файлов (остатки Полоцк/Липковская, КД, рецепты) хранятся
в Postgres-таблице `document_archive` как base64 в текстовом поле.

**Why base64 в Postgres, а не объектное хранилище:** в проекте нет настроенного
object storage; файлы небольшие (Excel/КД — десятки КБ, лимит загрузки 30 МБ),
Postgres уже подключён и общий для всех пользователей прода. Минус — раздувание
~33% и копии в памяти; при росте объёма мигрировать на bytea/object storage,
оставив в БД только метаданные.

**«Действующий документ» и отметка времени:** действующий = последний по
`uploadedAt` (default now() в БД). Время форматируется на фронте в фиксированной
зоне Europe/Moscow — поэтому одинаково для всех и переживает перезагрузку (берётся
из БД, не из клиента).

**Авторизация:** в проекте `AUTH_DISABLED = true` (server/src/auth/middleware.ts) —
`requireAuth` пропускает всех как dev-admin. На data-роутерах (dashboard, recipes,
documents) он всё равно навешан как защита на будущее, когда вход вернут. Загрузка
(upload) и raw-materials — публичные. Скачивание файла идёт через
`window.location.href` (как экспорт дашборда) — без Bearer-заголовка; когда вход
вернут, скачивание тоже придётся доработать.

**Кодировка имён файлов:** multer отдаёт имя из multipart как latin1 — кириллица
приходит «кракозябрами». Перекодировать latin1→utf8 только когда в строке есть
байты 0x80–0xFF и результат без U+FFFD; правильная кириллица (U+0400+) в этот
диапазон не попадает, поэтому не портится. Применять и при сохранении, и при
чтении (чинит уже сохранённые битые записи без миграции), и в Content-Disposition.

**Схема в проде:** post-merge скрипт `scripts/post-merge.sh` делает
`drizzle-kit push --force` (неинтерактивно). Ручной `npm run db:push` без --force
зависает на TTY-промпте — для новой таблицы проще создать через psql напрямую.

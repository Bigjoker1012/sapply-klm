# Очередь распознавания: что показывать

Очередь нераспознанных позиций (лист ReviewQueue) питается из загрузок. Два правила
фильтрации, оба durable-решения:

## Глобальные постоянные исключения «не сырьё»
- Кнопка «не сырьё» на главном экране НЕ просто резолвит строки очереди — она пишет
  название в отдельный лист Excluded (нормализованный текст = lowercase+trim+схлопнутые
  пробелы) и закрывает все строки очереди с этим текстом.
- При любой будущей загрузке исключённые имена не попадают в очередь (фильтр на入).
- **Why:** раньше «не сырьё» лишь резолвило строки → при повторной загрузке того же
  файла мусор возвращался. Пользователь хочет «исключил навсегда».
- **How to apply:** при добавлении в очередь и при чтении очереди всегда сверяться с
  Excluded по нормализованному имени. Отдельная вкладка «Исключённые» — только просмотр.

## Фильтр похожести — ТОЛЬКО для КД Липковской
- КД (ведомость 1С) содержит хозтовары/запчасти/тару — не сырьё. Поэтому перед
  постановкой в очередь имена из КД прогоняются через гейт похожести на эталонный
  корпус: справочник (full+short) + синонимы (Aliases) + амбарка Полоцка (PlantStock
  name) + компоненты рецепта (RecipeLines name).
- Гейт: значащие токены (significantTokens — тара/единицы/числа отброшены); оставляем,
  если есть общий характерный токен длиной >=4 ИЛИ хотя бы 2 общих токена. НЕ использовать
  tokenMatchScore (subset) — он рассчитан на авто-привязку, для гейта слишком строг.
- **Why:** рецепт и амбарку Полоцка пользователь просил НЕ трогать (там мусора нет);
  фильтр включён только в КД-хендлере, не в общих /unmatched.
- **How to apply:** любой новый источник с мусором гейтить отдельно; не выносить фильтр
  глобально, иначе отсеются законные новые позиции из чистых источников.

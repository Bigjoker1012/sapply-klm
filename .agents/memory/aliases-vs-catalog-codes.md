# Лист Aliases и каталог используют РАЗНЫЕ схемы кодов

Распознавание сырья читает каталог из Google Sheets «Syryo» и синонимы из листа
«Aliases». Каталог кодирует позиции как `RAW_001..RAW_068` (с подчёркиванием,
сплошная нумерация). Лист Aliases смешивает два формата строк:
- NEW (авто, записан кодом): `A=AL_xxx, B=raw_uid, C=текст_синонима, D=источник`
- OLD (курируемый вручную): `A=код, B=текст_синонима, C=тип, D=источник`

## Главная ловушка: коды НЕ соответствуют по номеру
Курируемые коды (`RAW001`, `RAW020`, `RAW030`…) — из СТАРОЙ схемы нумерации
каталога с другим составом/порядком. Совпадают только первые три витамина
(A, D3, E). Дальше расходятся: `RAW030 «Сульфат марганца»` → текущий каталог
`RAW_030` = «Биогром Холин»; `RAW020 «Оксид магния»` → `RAW_020` = «кобальт у/к».

**Правило:** НИКОГДА не «нормализовать подчёркивание» (RAW030→RAW_030) для
резолва синонима к каталогу. Это даёт уверенно-неверные привязки — для
снабжения опаснее, чем отсутствие привязки (можно заказать не то вещество).
Код считать валидным только при ТОЧНОМ совпадении с кодом каталога.

**Why:** обнаружено прямым сравнением: после нормализации ~3 верных и ~20+
неверных привязок.

**How to apply:** см. `parseAliasRows` в `server/src/services/sheetsService.ts` —
exact-match по validUids, `resolved=false` для несовпавших.

## Вторая порча: авто-синонимы (AL_) частично неверны
99 из 104 AL_-строк ссылаются на валидные коды каталога, НО часть привязок
смыслово ошибочна, потому что их записал старый багованный матчер. Пример:
`AL_… , RAW_011, "Сульфат марганца 32%"` — марганец привязан к «сульфат магния».

**Вывод:** надёжны только ТЕКСТЫ синонимов. И курируемые коды, и авто-коды
ненадёжны. Корректное лечение — перепривязка текстов синонимов к каталогу
(ИИ-сопоставление с подтверждением), а не правка кодов.

## Безопасное правило применения ИИ-перепривязки
gpt-4o-mini хорошо чинит явные ошибки (марганец→магний), НО иногда снимает
ВЕРНУЮ привязку, не найдя дословного совпадения (напр. «Биогром Энерджи»→нет,
хотя есть `RAW_029 Биогром Энерджи`).

**Правило:** применять предложение ИИ к строке только если есть uid И уверенность
high/medium; иначе сохранять текущий код строки. НИКОГДА не затирать валидную
resolved-привязку пустотой/no-match.
**Why:** для снабжения потеря верной привязки = недозаказ позиции; слепое
применение ИИ регрессирует 3-4 корректные строки.
**How to apply:** ИИ-сопоставление батчами надёжно; одиночный вызов
`aiMatcher.suggestMatches` возвращает [] (форма ответа). Всегда показывать CSV на
сверку пользователю и писать в лист Aliases только после подтверждения.

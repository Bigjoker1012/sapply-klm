---
name: Парсинг рецепта PDF — текстовый слой раньше OCR
description: Почему PDF рецепта (КК-…) надо парсить текстовым слоем, а OCR — только фолбэк
---

PDF рецептов («Рецепт комбикорма», КК-…) — это ЦИФРОВЫЕ PDF с чистым текстовым
слоем, а не сканы. Парсить их надо так: СНАЧАЛА текстовый слой (`pdf-parse`),
OCR (PyMuPDF render + tesseract) — только фолбэк для настоящих сканов.

**Why:** чисто-OCR-путь падал на этих PDF с `MuPDF error: No common ancestor in
structure tree` — fitz спотыкается о дерево структуры документа. Хуже того, при
падении python-скрипт писал в stdout не-JSON, и `JSON.parse` на стороне Node
кидал исключение → пользователь видел «не разбирает pdf». Текстовый слой у этих
файлов есть всегда, OCR для них не нужен и только ломает.

**How to apply:** при жалобах «не читает/не разбирает PDF» — сначала проверить,
есть ли текстовый слой (`pdf-parse`), и только при его отсутствии звать OCR.
Ловушка кириллицы в regex: `\b` НЕ срабатывает после кириллической буквы (н-р
после «т» в «Выработки») — границы слова `\b` в JS не дружат с кириллицей,
писать паттерны без `\b`.

**Важное исключение (премиксы Полоцкого КХП, «РЕЦЕПТ ПРЕМИКСА Д-…/Б…/ПЛЦ-…»):**
эти PDF из 1С НЕ имеют извлекаемого текстового слоя вообще — текст вшит кривыми,
и `pdf-parse`, и poppler `pdftotext -layout` возвращают пусто (~1-2 символа).
Значит единственный путь — OCR, а OCR этой ПЛОТНОЙ многоколоночной таблицы
(Активн|%ввода|Норма г/т|Расход кг|Цена|Стоим + столбцы качества справа)
НЕНАДЁЖЕН: парсер `ocr_recipe.py` берёт только строки, начинающиеся с `[`
(tesseract так размечает не все строки) → из ~19 компонентов вытаскивает ~4.
**Why:** снять фильтр `[` нельзя — тогда лезет мусор (`+375`, «ДАТА ПЕЧАТИ») и,
опаснее, неверные количества (СЕЛЕНИТ НАТРИЯ 9% вместо 0,45% — ×20), что для
системы закупок хуже явной ошибки.

**Решение (внедрено): AI-vision вместо OCR для таких PDF.** Цепочка фолбэков в
`parseRecipePdf`: текстовый слой → vision (`parseRecipeWithVision` в aiMatcher.ts:
рендер страницы в PNG через poppler `pdftoppm` → отправка картинки в модель с
vision, строгий JSON) → tesseract как последний фолбэк. На ПЛЦ-164 vision дал 17
компонентов с корректными количествами (расход = % × выработка, без искажений),
OCR давал 4. **Why:** vision читает плотную многоколоночную таблицу-картинку
надёжно, в отличие от построчного tesseract. Tesseract оставлен фолбэком на
случай отсутствия ключа OpenAI / сбоя сети.
**Подводные камни, на которые ловились:** (1) клиент OpenAI инициализировать
ЛЕНИВО — конструктор `new OpenAI()` бросает без ключа и ломает фолбэк в OCR;
(2) temp-файлы рендера держать в `mkdtemp`-директории на запрос и сносить всю
директорию в finally — иначе коллизии имён при параллельных загрузках и утечки
PNG. Альтернатива без ключа — грузить рецепт как Excel (`parseRecipeExcel`).

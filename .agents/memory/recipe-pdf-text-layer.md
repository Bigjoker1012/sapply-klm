---
name: Парсинг рецепта PDF — текстовый слой раньше OCR
description: Почему PDF рецепта (КК-…) надо парсить текстовым слоем, а OCR — только фолбэк
---

PDF рецептов («Рецепт комбикорма», КК-…) — это ЦИФРОВЫЕ PDF с чистым текстовым
слоем, а не сканы. Парсить их надо так: СНАЧАЛА текстовый слой (`pdf-parse`),
OCR (PyMuPDF render + tesseract) — только фолбэк для настоящих сканов.

**Why:** чисто-OCR-путь падал на этих PDF с `MuPDF error: No common ancestor in
structure tree` — fitz спотыкается о дерево структуры документа. Хуже того, при
падении python-скрипт писал в stdout не-JSON, и `JSON.parse` на стороне Node
кидал исключение → пользователь видел «не разбирает pdf». Текстовый слой у этих
файлов есть всегда, OCR для них не нужен и только ломает.

**How to apply:** при жалобах «не читает/не разбирает PDF» — сначала проверить,
есть ли текстовый слой (`pdf-parse`), и только при его отсутствии звать OCR.
Ловушка кириллицы в regex: `\b` НЕ срабатывает после кириллической буквы (н-р
после «т» в «Выработки») — границы слова `\b` в JS не дружат с кириллицей,
писать паттерны без `\b`.
